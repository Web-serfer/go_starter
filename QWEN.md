# Правила взаимодействия с Qwen Code Assistant

## Основные принципы

1. **Изменения в коде возможны только с явного разрешения пользователя**

   - Перед внесением любых изменений в файлы ассистент должен получить подтверждение от пользователя
   - Все изменения должны быть предварительно объяснены пользователю
   - Перед решением конкретной задачи всегда составлять план
   - После внесения изменений в код - проводить проверку - только после этого приступать к дальнейшему решению задачи

2. **Прозрачность действий**

   - Ассистент должен объяснить, какие изменения планируется внести
   - Необходимо указать, в какие файлы будут внесены изменения
   - Следует объяснить последствия предполагаемых изменений

3. **Безопасность кода**

   - Все изменения должны проходить проверку на безопасность
   - Не должны вноситься изменения, которые могут повредить функциональность приложения

4. **Согласование архитектурных решений**

   - При внесении изменений, затрагивающих архитектуру приложения, необходима дискуссия с пользователем
   - Предложения по улучшению архитектуры должны обсуждаться до реализации

5. **Работа с разными типами проектов**

   - Уважать существующую архитектуру и стиль кода проекта
   - Следовать установленным в проекте принципам и паттернам

6. **Язык общения**

   - Всё общение с пользователем происходит на русском языке

7. **Проверка изменений**

   - После внесения изменений в код не требуется запускать сервер разработки для проверки
   - Пользователь самостоятельно запускает сервер и проверяет изменения
   - никогда не запуская исполнение кода командой run go main.go

8. **Выбор лучшего синтаксиса для Go**

   - При создании кода на Go всегда использовать наиболее идиоматический и элегантный синтаксис
   - Следовать принципам и стилю, принятым в сообществе Go (Effective Go)
   - Использовать группировку объявлений переменных (var (...) ), когда это уместно
   - Предпочитать короткое объявление переменных (:=) там, где это возможно

9. **Дополнительные правила для Go**

   - Использовать именованные возвращаемые значения в функциях, когда это улучшает читаемость
   - Следовать соглашениям об именовании (camelCase для экспортируемых и неэкспортируемых идентификаторов)
   - Использовать интерфейсы для абстракции и создания гибкого кода
   - Обрабатывать ошибки по Go-идиомам, не игнорировать возвращаемые ошибки
   - Использовать defer для очистки ресурсов и обеспечения корректного завершения функций
   - Применять принципы clean code: функции должны быть компактными и выполнять одну задачу
   - Использовать go fmt для форматирования кода и go vet для проверки потенциальных ошибок
   - Писать тесты для обеспечения качества кода и предотвращения регрессий
   - Документировать код: использовать комментарии для пакетов, функций и структур в соответствии с Go-конвенциями
   - Работать с указателями осознанно: понимать разницу между значениями и указателями, использовать указатели только когда это необходимо
   - Использовать конкурентность правильно: применять goroutines и channels для эффективной параллельной обработки
   - Оптимизировать использование памяти и следить за производительностью: избегать ненужных копирований и утечек памяти
   - Структурировать проект по Go-конвенциям: использовать правильные имена пакетов и директорий
   - Использовать логирование по Go-идиомам: применять подходящие библиотеки логирования и форматы
